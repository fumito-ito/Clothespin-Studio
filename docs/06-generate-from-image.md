## 1. 何をやろうとしたか

「任意の入力から、それに似せた**立体**構造物をピンで組んで提示する」機能。

ユーザーとの議論で次の整理に至った:

- **平面化は却下**。洗濯バサミ工作の価値は立体構造にあるため、画像をそのまま並べる
  モザイク（旧 Stage 1）や、輪郭をなぞる線画（旧 Stage 2）は「連結はするが幾何的に平面」で不採用。
- **1 枚の画像には奥行き情報が無い**のが根本制約。立体を出すには「奥行きの作り方」を決める必要がある。
- そこで 2 路線に再定義し、**A→B の段階実装**を選択（ユーザー決定）:

| | **A. レリーフ（画像→高さマップ）** | **B. 3D モデル入力（ボクセル化）** |
| --- | --- | --- |
| 入力 | 画像 | GLB / STL など 3D モデル |
| 奥行きの出どころ | 輝度→高さに変換 | モデル自体が保持 |
| 出力 | 地面に林立する**連結タワー群**の起伏（上から見ると絵、横から見ると地形） | 任意立体をピンのタワー/格子で充填した**本物の 3D** |
| 実装規模 | 小 | 中（モデル読込 + ボクセル化） |

**設計の肝**: A と B は「**タワー生成器**」という核を共有する。A を先に作れば、B 固有なのは
「3D モデル読込 + ボクセル化（→ セル格子）」だけになり、その先のタワー生成・描画・部品表・
エクスポートは既存資産がそのまま使える。

旧「Stage 3（画像の深度推定 ML で立体化）」は、ブラウザでの ML 実行が重く品質も不安定なため、
B が動くなら不要という結論。

## 2. 核心の発見：タワーの作り方（計測で確定）

ピンを積んでタワーにする連結方法を実測で確定した。**重要: クロソピンの連結幾何では「完全な
垂直スタック」は作れず、どの連結でも高さに比例して横へドリフトする**（当初「g0/g1 交互でほぼ直立」
と考えたが、計測では交互は相殺せずむしろドリフト増だった）。

- **ルートピンを鼻先下向きに立てる**: 姿勢 `R_y(+90°) = [0, √½, 0, √½]`、接地高さ `z = 31mm`。
- **以降は `g0`（上ハンドル先端）を連続連結**。計測で最も背が高く（約 59mm/段）、**面内**
  （厚み Y≈12mm を保つ）でドリフトも比較的小さい。
- 不採用（計測で確認）:
  - `g0`/`g1` 交互 → ドリフト増（footX が大）。
  - `g2`/`g3` 連続 → 段がほぼ重なり積み上がらない。
  - リング連結（`g4`/`g6`）→ 面外に螺旋（fy が大）。
- **間隔（衝突回避）**: 全タワーが同じ向き・同量で傾く（平行）ため、隣接タワーの X 間隔は全高さで
  一定（= pitchX）。よって必要なのは累積ドリフトではなく「各段 1 ピン分の幅」だけ。実測で
  **pitchX ≥ 55 / pitchY ≥ 18mm なら高さに依らず干渉 0**。余裕を見て既定 `60×24mm` を採用。
  → レリーフはコンパクト（例: 3300 ピンで約 129×119×35cm）で**干渉判定（main 由来）でも赤が出ない**。

この知見は [`src/domain/generator.ts`](../src/domain/generator.ts) 冒頭コメントにも記載済み。

### データモデルへの影響
- **なし**。生成結果は通常の自由ピン（ルート）+ 連結ピンの集合なので、保存/読込・部品表・
  GLB/STL/PNG エクスポートはすべてそのまま機能する想定。

### レリーフの向き（重要）
- 画像の列→ドメイン X、行→ドメイン −Y で地面（X-Y 平面）に配置。各ピクセルが +Z に伸びるタワー。
- **真上から**見ると量子化された絵、**横から**見ると高さの起伏（地形）になる。
- `DEFAULT_RELIEF_PARAMS = { pitchX: 60, pitchY: 24 }`（mm・固定）。`imageToCells` で
  `pitchX/pitchY` 比により行数を補正し、真上ビューが歪まないようにしている。
