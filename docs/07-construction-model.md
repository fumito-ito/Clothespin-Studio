# 07. 立体構築モデルの調査と設計（ルールベース生成アセンブリ）

> ステータス: 設計検討（`feature/generate-from-image` ブランチ）。
> 「画像/3D から **連結された組み立て可能な** 立体をクロソピンで作る」ための構築モデルを定める。
> 実装の前に、参考になる構築玩具の方式を調査して方針を確定した記録。

## 1. 背景と問題

[06](06-generate-from-image.md) のレリーフ生成は **各タワーが独立（フォレスト）** で、相互に連結
していない＝1 つの組み立て可能体になっていない。連結しようとすると、クロソピンの連結幾何の制約に
ぶつかる（実測で確定）:

- **回転 0 の単一連結は存在しない**。最小でも 1 リンクあたり 31°（`g1`）回転し、z 方向にもズレる。
- → 連結チェーンは必ず曲がる/浮く。**規則正しい平面格子も垂直タワー列もタイル状に敷けない**。
- period-2（2 連結）で回転を打ち消す「直線ユニット」は作れる（例 `g4/g6` 系で `(-40,±30,±10)`）が、
  素朴に格子化すると **自己干渉（試作 36/49）・非平面（厚み 119mm）** になった。

結論として、**「格子に置く」発想ではクロソピンの立体は作れない**。専用の構築モデルが要る。

## 2. 調査：類似玩具・手法の構築モデル

| 玩具/手法 | モデル | クロソピンへの適合 |
| --- | --- | --- |
| **LEGO**（BrickLink Studio / legolization） | ボクセル化 → 層ごとにブロック充填 + 連結/安定性解析（GA・力解析） | ✗ **規則格子に乗るコネクタ前提**。クロソピンは格子に乗らない |
| **ハブ-ストラット**（Zometool / K'NEX / Posey） | ノード（多方向ソケット）＋ストラットのグラフ。角度は**ハブ**が規定 | △ クロソピンは「クランプ付きストラット」だが**専用ハブが無く**、角度がクリップに焼き込まれている |
| **磁性パネル**（Magformers） | 多角形ネットを辺で連結 → 折って多面体 | ✗ 面要素前提。クロソピンは線/クリップ要素 |
| **インターロッキング・アセンブリ / シェイプ・グラマー** | **離散要素の言語＋連結ルール**で形を創発させ、目標形状にフィット | ◎ **クロソピンに最も合うパラダイム** |

出典は §8。

## 3. 結論：採用パラダイム

**ルールベース生成アセンブリ（シェイプ・グラマー / インターロッキング・アセンブリ）** を採用する。

理由: クロソピンは **向きを持つクリップ・コネクタ**で、連結が **有限のルール集合（catalog）** で
記述でき、格子に乗らない。したがって「格子に置く」のではなく「**ルールで成長させて目標形状を
近似する**」のが本質的に正しい。LEGO の貢献は「**目標のボクセル化**」だけ借用する。

## 4. 提案モデル

### 4.1 連結カタログ（= 文法の語彙）
- 有効な `(socket, roll, pitch) → 相対変換（親ローカル）` の有限集合。
- すでに `domain/solve.ts` の `childWorldMatrix` で計算可能（[06](06-generate-from-image.md) の period-2 探索が実例）。
- 各エントリは「子の相対位置・相対回転・占有ソケット」を持つ。これが文法の生成規則。

### 4.2 目標表現（target）
- 望む形を **ボクセル化**する（画像→高さ場、または 3D モデル→ボクセル）。
- ボクセルの一辺 ≒ ピンの代表寸法/間隔。各ボクセルに「色（最近傍パレット）」を持たせる。

### 4.3 生成：最良優先で連結木を成長
1. 種ピン（1 本）を目標内に置く。
2. フロンティア（未占有ソケットを持つ既存ピン）から、カタログの各ルールで子候補を列挙。
3. 各候補をスコア評価:
   - (a) **被覆**: 未充填の目標ボクセルに入るか（入る数が多いほど良い）。
   - (b) **干渉なし**: 既存ピンと衝突しない（`domain/collision.ts` の OBB×SAT を流用）。
   - (c) **逸脱なし**: 目標ボクセルの外へ大きくはみ出さない。
   - (d) 連結は木構造なので**常に保たれる**（各ピンは親 1 つ）。
4. 最良候補を採用してピンを追加。目標を覆う or 有効手が尽きるまで反復。

### 4.4 制約・評価
- **連結**: 木（フォレスト→単一ルート）で保証。
- **干渉**: 既存の干渉判定をハード制約に。
- **安定性**（将来）: LEGO 研究の力解析を参考に、自重で倒れない/接合が持つかを後段で評価。
- **被覆率**: `覆えた目標ボクセル / 全目標ボクセル`。これが品質指標。

## 5. 既存資産との対応

| 必要物 | 既存 / 新規 |

| 連結カタログ生成 | `solve.childWorldMatrix` + `clothespin` の socket 定義（流用） |
| 干渉判定 | `domain/collision.ts`（流用） |
| 連結整合性 | `domain/graph.ts`（流用） |
| 目標ボクセル化（画像→高さ場） | `io/imageToCells.ts` を拡張（高さ→ボクセル列） |

データモデル（[03](03-data-model.md)）は変更不要。生成結果は通常のピン木。

## 6. 限界・未知数（正直な見立て）

- インターロッキング系は研究でも「**組み立て方向に制約**」があるとされ、**カタログが届かない領域は
  カバーできず近似**になる。完全再現は保証されない。
- カタログの相対変換は離散・固定なので、**目標ボクセルに綺麗に整列するとは限らない**（成長が逸れる）。
- よって「どこまで似せられるか」は**被覆率を実測**して見極めるのが先決（§7）。
